# 任务二：知识图谱问答智能体 — 技术报告

***

## 1. 概述

任务二目标是构建一个**医疗知识图谱问答智能体**。给定一段医疗文本，智能体能自动完成实体识别→关系抽取→三元组生成→验证→回答的全流程。核心是**3个纯规则引擎算子** + **Nexent双代理架构**（后合并为内置自查的单体v3.0），零外部LLM依赖。

### 核心竞争力

| 维度      | 说明                               |
| :------ | :------------------------------- |
| **轻量**  | 纯规则引擎，词典+模式匹配，零深度学习依赖            |
| **可解释** | 每个判定都可追溯——什么词典命中了、什么触发词匹配了       |
| **快速**  | 三算子端到端延迟 < 1ms（1000条评测 33.5μs/条） |
| **安全**  | 内置直接冲突+禁忌冲突双重检测                  |

## 3. Schema设计（v2.5 — 最终版）

### 实体类型（4种）

| 类型 | 英文          | 示例             | 匹配方式        |
| :- | :---------- | :------------- | :---------- |
| 疾病 | disease     | 2型糖尿病、高血压、冠心病  | 词典精确匹配+后缀识别 |
| 症状 | symptom     | 头痛、发热、多饮多尿     | 词典精确匹配      |
| 药物 | drug        | 二甲双胍、阿司匹林、胰岛素  | 词典精确匹配      |
| 检查 | examination | 血常规、CT、心电图、MRI | 词典匹配+缩写识别   |

### 关系类型（4种）

| 关系     | 说明       | 示例           | 触发模式               |
| :----- | :------- | :----------- | :----------------- |
| **导致** | 疾病→症状    | 糖尿病→导致→多饮多尿  | "症状"、"表现"、"引起"等触发词 |
| **治疗** | 药物→疾病    | 二甲双胍→治疗→糖尿病  | "治疗"、"服用"、"用药"等触发词 |
| **用于** | 检查/辅助→疾病 | 血常规→用于→糖尿病检查 | "检查"、"检测"、"诊断"等触发词 |
| **禁忌** | 疾病/药物→禁忌 | 高血压→禁忌→麻黄碱   | "禁忌"、"禁用"、"不宜"等触发词 |

### Schema演进

```
v1.0（4实体+3关系）→ v2.0（4实体+3关系+上下文约束）→ v2.5（4实体+4关系+禁忌关系）
```

## 4. 核心算子

### 4.1 kg\_entity\_recognizer — 实体识别

```
输入: "糖尿病患者视力模糊，应该用什么药？"
步骤:
  1. 分句 →  ["糖尿病患者视力模糊", "应该用什么药"]
  2. 词典匹配 →  疾病词典命中"糖尿病" → 症状词典命中"视力模糊"
  3. 去重 →  最终实体: [{type:"疾病",name:"糖尿病"}, {type:"症状",name:"视力模糊"}]
输出: {"entities": [...], "total_count": 2}
```

关键技术：

- **首字哈希索引**：O(1)词典查找（不是逐词遍历）
- **词典量**：60+疾病、20+症状、30+药物、20+检查和治疗
- **上下文过滤**：如"高血压≠血压高"、过滤"副作用"等非实体短语

### 4.2 kg\_relation\_extractor — 关系抽取

```
输入: "糖尿病患者视力模糊"
步骤:
  1. 句内共现分析 → "糖尿病"和"视力模糊"在同一句
  2. 触发词匹配 → "糖尿病"+"视力模糊"之间无明确动词 → 走共现推理
  3. 关系判定 → 疾病→导致→症状
输出: [{head:"糖尿病", relation:"导致", tail:"视力模糊"}]
```

两种推理通道：

| 通道       | 方式                 | 适用场景      |
| :------- | :----------------- | :-------- |
| **模式匹配** | 触发词表匹配（"治疗"→治疗关系）  | 含明确触发词的句子 |
| **共现推理** | 同一句内实体对→基于实体类型推断关系 | 无明确动词的句子  |

**跨句隔离**：默认禁跨句推理（防止幻觉），但短距离内（1句）可配置。

### 4.3 kg\_triple\_generator — 三元组生成+冲突检测

```
输入: entities + relations
步骤:
  1. 组合为 (head, relation, tail) 三元组
  2. Schema校验 → 检查实体类型是否匹配关系定义
  3. 直接冲突检测 → 同实体不同关系（如"治疗"和"禁忌"同时存在）
  4. 禁忌冲突检测 → 药物A和药物B是否存在禁忌关系
  5. 去重 → 相同三元组合并
输出: {"triples": [...], "conflicts": [...]}
```

| 检测类型 | 示例                    | 严重程度 |
| :--- | :-------------------- | :--- |
| 直接冲突 | 糖尿病→治疗→A药 + 糖尿病→禁忌→A药 | 🔴 高 |
| 禁忌冲突 | 阿司匹林 + 华法林 联用         | 🟡 中 |

## 5. 智能体设计与Prompt

### 版本演进

| 版本       | 架构          | 特点                                 | 状态           |
| :------- | :---------- | :--------------------------------- | :----------- |
| v2.5     | 双代理（推理+验证）  | Supervisor模式，managed\_agents: \[2] | ⚠️ 验证代理触发不稳定 |
| **v3.0** | **单体+内置自查** | 5步流程，内置5条验证规则                      | ✅ **推荐使用**   |

### v3.0 Prompt核心

**duty\_prompt**：角色定义 + 3个MCP工具说明 + 五步处理流程

```
1. knowledge_graph_search — 实体识别
2. entity_recognition — 关系抽取
3. relation_extraction — 三元组生成
```

**constraint\_prompt**（8条规则）：

```
1. 工具调用格式：必须用 <RUN> 代码块
2. params key必须双引号
3. 禁止模拟结果
4. 三步强制调用
5. 自查验证（输出前必做）
6. 回答格式：自然语言+表格
7. 错误处理策略
8. 自查5项：实体准确性/关系一致性/冲突检测/安全性/信息标注
```

## 9. 创新点

1. **纯规则引擎的极速KG抽取** — 三算子端到端33.5μs/条，比基于LLM的管线快1000倍以上，且输出100%可复现
2. **双通道关系推理** — 触发词匹配 + 共现推理双通道互补，有触发词走精确匹配，无触发词走类型推断
3. **两级冲突检测** — 直接冲突（A治疗且禁忌B） + 禁忌冲突（A药+B药联用风险），确保知识图谱逻辑一致性
4. **内置自查验证** — 输出前5项自检（实体/关系/冲突/安全/标注），不依赖不稳定的双代理触发机制
5. **首字哈希索引** — O(1)词典查找，1000条文本端到端仅33ms
